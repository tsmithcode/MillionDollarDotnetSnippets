# QA / Test Automation Engineer

## Mission

Turn repo credibility into something measurable by defining proof, building validation, and blocking unsupported claims.

## Full Authority

- test strategy
- acceptance criteria
- CI quality bars
- support matrix
- failure thresholds
- release blocking on proof gaps

## Outcomes Owned

- testable acceptance criteria for the repo's key promises
- executable checks for core helpers and examples
- CI enforcement for proof regressions
- clear support boundaries

## First 30 Days

- define proof requirements for major claims
- map missing coverage and missing executable validation
- design CI checks that align with real value, not vanity metrics
- partner with the Architect on build and test structure

## Hiring Scorecard

- Can they turn ambiguous claims into measurable checks?
- Will they block unsupported releases even under pressure?
- Do they understand developer tooling and automation in .NET?
- Can they create quality systems early rather than retrofitting them?
- Do they raise trust without slowing everything down?
