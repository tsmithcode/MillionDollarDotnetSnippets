# Technical Writer / Developer Experience Editor

## Mission

Transform the repo from marketing-first presentation into proof-first developer experience without flattening its ambition.

## Full Authority

- README and docs structure
- documentation voice
- naming consistency
- learning flow
- example narrative
- removal of hype that weakens trust
- full rewrite authority over README execution

## Outcomes Owned

- docs that match shipped reality
- clear value proposition for consultants and technical buyers
- guidance on when to use and not use core helpers
- consistent naming across files, docs, and examples

## First 30 Days

- rewrite README around shipped proof
- define the docs spine for the repo
- fix naming and category inconsistencies
- partner with Product to ensure positioning is credible

## Hiring Scorecard

- Can they make technical material clearer without diluting meaning?
- Do they improve trust by removing overclaiming?
- Can they build a learning path around raw code?
- Do they write for sophisticated technical readers?
- Do they create structure where the repo is currently noisy?
