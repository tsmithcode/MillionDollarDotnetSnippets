# Story Framework Wizard

## Product direction

The public website should feel less like a traditional marketing site and more like a guided framework wizard.

The user should feel like they are being walked through a serious operational system:

- clear
- calm
- accountable
- step-based
- easy to trust

The closest inspiration is not a flashy SaaS homepage.
It is a high-accessibility civic workflow such as a U.S. voting or poll-system experience where the interface must be understandable under pressure and usable by people with widely different levels of technical confidence.

## Visual thesis

Executive-grade product guidance with civic clarity.

The experience should feel:

- decisive instead of decorative
- structured instead of crowded
- accessible instead of trendy
- premium because it is rigorous, not because it is loud

## Wizard structure

### Step 1: Who are you

Help the visitor self-identify:

- consultant
- solution architect
- engineering lead
- operations-minded builder
- enterprise integration owner

### Step 2: What pain are you solving

Offer a small number of high-ROI entry problems:

- I need to ship internal tools faster
- I need to make integrations reliable
- I need a rules and validation foundation
- I need reusable building blocks across client work

### Step 3: What proof do you need

Let the user choose the confidence layer they want:

- example first
- architecture first
- diagnostics first
- API surface first

### Step 4: Show the path

Guide them into:

- the golden-path quickstart
- the capability-first API
- the founder/about story
- the use cases for their persona

### Step 5: Prove it in-browser

Show:

- real browser/device-tested workflows
- visible success markers
- accessibility notes
- expected output

## Accessibility requirements

- keyboard-first flow
- large tap targets
- obvious progress indicator
- explicit allowed/blocked reasoning
- no hidden state changes
- strong linear heading structure
- no SVG-only explanation of critical meaning
- every important decision visible in text

## Content rules

- do not overwhelm the first-time user
- one question per step
- one action per screen
- one clear “next” button
- explain why a path is recommended
- expose diagnostics when something is blocked

## Product consequence

This shift makes the framework easier for zero-knowledge users without weakening the expert story.

It also turns the public site into a usable part of the product itself:

- orientation
- qualification
- onboarding
- proof
- conversion
