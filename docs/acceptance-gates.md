# Acceptance Gates

This repo does not ship on aesthetics alone.
Changes are accepted when they strengthen trust and measurably improve the asset.

## Repo-Level Gates

- The repo title, README, structure, and shipped content must describe the same thing.
- Every major section must have a named domain owner.
- No advertised category may exist only in marketing copy.
- High-impact claims must point to code, examples, tests, or documented proof.

## Technical Gates

- The codebase must be buildable from a documented starting point.
- Public APIs and helpers must be grouped coherently, not left as loose draft files.
- Duplicated or misleading content must be removed or replaced.
- Example usage must exist for the most important public helpers.

## Documentation Gates

- The README must describe what exists today, not what might exist later.
- Naming across files, modules, docs, and examples must be consistent.
- Every promoted capability must explain when to use it and when not to use it.
- Documentation must reduce ambiguity for a technical buyer or consultant evaluating the repo.

## Quality Gates

- Tests or executable checks must cover the repo's core behaviors.
- Acceptance criteria must exist for every major advertised capability.
- CI should fail if proof is broken, missing, or inconsistent with docs.
- QA can block release when support claims exceed validated coverage.

## Positioning Gates

- Commercial language must be supportable by visible evidence.
- The repo promise must stay ambitious without overstating maturity.
- Audience, offer, and value proposition must remain clear after major changes.
- The README and vision may change radically if the new version better expresses the title and target ROI.

## Identity and GitHub Gates

- The first screen of the README must make the target persona obvious.
- The repo must have a distinct visual identity, not default GitHub plainness.
- Community health files, issue templates, and CI must signal serious stewardship.
- The repo structure must read like a product company artifact, not a loose archive.
- The inherited README structure must never limit a better product narrative.

## Accessibility Gates

- Product-facing SVGs must include descriptive text metadata.
- Accessibility must be reviewed as part of premium design work, not after it.
- The fastest path to first success must remain readable and obvious.
- Visual polish must not create ambiguity, low contrast, or inaccessible hierarchy.

## Autonomy Gates

- Each domain owner must be able to make at least one major change without waiting on upstream approval.
- Ownership boundaries should speed decisions, not create veto chains.
- Conflict resolution must happen through named decision rights, not informal politics.
